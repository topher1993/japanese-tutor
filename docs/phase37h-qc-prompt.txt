## What the user reported
User says: "there is an issue on the todo feature, when accessed the week1
lesson thru todo i cant mark the lesson complete". The weekly-todo board
renders on the Home and Lessons screens (gated behind a dev-only flag)
and shows lesson/flashcard/daily-rush todos per week. When a learner
taps the "Open" CTA on the lesson todo, the lesson detail view opens.
Tapping "Mark this lesson complete" advances the lesson, but the
on-screen todo board does NOT update — it stays at "0 / 5 lessons".
This is the visible symptom the user described as "I can't mark the
lesson complete".

## What the fix did
Two bugs:

1. weeklyTodoService.statusForTodo derived `isLegacyWeek` purely from
   `!isInitialized`, so even the current week showed "Completed before
   weekly todos were introduced" on every todo row. Now it threads
   `currentWeek` through and only treats the row as legacy when
   `weekNumber < currentWeek && !isInitialized`.

2. The store's completeCurrentLesson only persisted todoStates via
   saveExtendedProgress. On the web runtime path, the repo is the
   in-memory learning repo which deliberately omits
   saveExtendedProgress, so the cache update was skipped entirely.
   The in-memory `progress` state in LessonsScreen/HomeScreen never
   reflected the recomputed todoStates, so the on-screen todo board
   always rendered with the empty initial payload. Now all five
   record* methods always update the in-memory `extendedCache`
   (visible via `store.getExtendedProgress()`), and only the disk
   persistence is gated on `typeof repo.saveExtendedProgress ===
   'function'`. Both LessonsScreen and HomeScreen now read
   `store.getExtendedProgress()` and feed the result into
   `todoPayload`, so the memoized payload reflects fresh state after
   the mark-complete handler's `setProgress(refreshed)`.
