## P0-1 â€” TSX generic-arrow parse failures â€” CLOSED âś…

**GPT-5.5 finding:** `App.tsx:117`, `App.tsx:176`, `src/services/learningContext.tsx:72` use `<T>(...)` generic arrow functions inside `.tsx` files. The Metro `.tsx` parser treats `<T>` as a JSX element-start and either fails to parse or silently strips the generic. Vitest does not exercise this path because ts-jest uses a separate channel. The error was confirmed by `npm run typecheck` failing with `JSX element 'T' has no corresponding closing tag`.

**Fix:** Replaced the inline generic-arrow with an IIFE cast to the interface method type:
```ts
getAllAsync: ((sql: string, ...params: unknown[]) =>
  db.getAllAsync(sql, ...(params as never[]))) as SqliteLikeDatabase['getAllAsync'],
```
The `SqliteLikeDatabase` interface defines `getAllAsync<T>(...)` itself, so the wrapper just forwards and the cast restores the generic. Added `import type { SqliteLikeDatabase }` to `App.tsx` (line 18).

## P0-2 â€” Real native reset path â€” CLOSED âś…

**GPT-5.5 finding:** The Settings "Reset all progress" affordance was a no-op on React Native. `clearOnboardingPreference()` only knew `window.localStorage`. `practiceProgressStore` had no `reset()`. `persistentSrsStore` had no `clearAll()`. `sqliteLearningRepository` had no `DELETE FROM` methods. **User-facing lie.**

**Fix (multi-file):**

1. `src/repositories/sqliteLearningRepository.ts` â€” added `deleteAllProgress(): Promise<void>` to the `PersistentLearningRepository` interface. The SQLite implementation issues `DELETE FROM progress` and resets the in-memory `progressCache` to `createInitialProgress('2026-06-18')`.
2. `src/repositories/inMemoryLearningRepository.ts` â€” added the same method to `LearningRepository`. In-memory impl just resets progress to initial state.
3. `src/services/practiceProgressStore.ts` â€” added `reset()` that delegates to `repo.deleteAllProgress()`. Also exported the `PracticeProgressStore` type alias (was a pre-existing TS error).
4. `src/services/persistentSrsStore.ts` â€” added `clearAll(): Promise<void>` to the `PersistentSpacedRepetitionScheduler` interface. SQLite impl issues `DELETE FROM kv_srs_cards`. In-memory impl clears its `allCards` array. Both handle missing-table gracefully by re-running `createTablesSql`.
5. `src/services/learningContext.tsx` â€” exposed `resetAll: () => Promise<{ srsRowsCleared: number }>` from the `LearningContext` value. Built via `makeResetAll(store, srs)` at every `setValue` call so the closure always sees the latest live stores. `resetAll` calls `Promise.all([store.reset(), srs.clearAll()])` and returns the SRS row count cleared (for UI confirmation).
6. `src/screens/SettingsScreen.tsx` â€” now consumes `resetAll` from `useLearningContext()`. Calls `await resetAll()` BEFORE `await onReset()` so the "Cleared N review cards" summary reflects what was actually persisted. Renders the summary via a new `lastResetSummary` state with `testID="settings-reset-summary"`.
7. `App.tsx` â€” `onReset` callback still calls `clearOnboardingPreference()` (web/native localStorage) + resets local React state (`setOnboarded(false)`, `setSupportLanguage('en')`). The new `resetAll` from the context handles SQLite state.

## P0-3 â€” FlashcardsScreen persistent SRS â€” CLOSED âś…

**GPT-5.5 finding:** `FlashcardsScreen.tsx:51` instantiates a fresh in-memory `createSpacedRepetitionScheduler()` per render via `useMemo`. `rateCard()` (lines 89-99) writes only to that scheduler. The persistent `srs` from `useLearningContext()` is never used. **Learner rates 50 cards â†’ all SRS state lost on cold start.** Same shape as Phase 22's P0-02, narrower scope.

**Fix:**

1. Removed `createSpacedRepetitionScheduler` import (line 15).
2. Removed `useMemo` import.
3. Removed local `const scheduler = useMemo(() => createSpacedRepetitionScheduler(), [])` (was line 51).
4. Destructured `srs` from `useLearningContext()` (line 36) alongside existing `ready, store`.
5. Added a re-hydration `useEffect` that depends on `[ready, srs]` and calls `srs.listCards()` + `srs.dueCount()` to populate local state from durable storage. This is the fix that makes cold-start persistence work â€” after a kill+relaunch, the previous session's SRS rows are still in SQLite and get re-loaded into React state.
6. Replaced local `srCards` state with `srsCardIdByRefId` (same shape) and added `persistedSrsCards` state that holds the durable snapshot. The `srCard` lookup now reads from this snapshot via `.find(c => c.id === srCardId)` instead of the in-memory mirror.
7. `rateCard` is now `async` and calls `srs.createCard(card.id)` / `srs.review(cardId, rating)` (the persistent versions that write through to SQLite). After rating, calls `await refreshSrs()` to re-read the durable snapshot so `dueCount` and `srCard` display update.
8. `dueCount` is now `useState<number | null>(null)` populated by `srs.dueCount()` (the SQLite `SELECT COUNT(*)` query, not the in-memory `scheduler.dueCards().length`).
9. Added a "Storage: persistent (durable)" / "in-memory fallback" line to the Card info disclosure so QA can visually confirm the backend in dev builds.
